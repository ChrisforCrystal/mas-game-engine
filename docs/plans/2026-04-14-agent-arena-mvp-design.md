# Agent Arena MVP 设计文档

日期：2026-04-14

状态：Draft v1

作者：Codex

## 1. 背景与目标

### 1.1 背景

团队希望设计一个内部使用的 Agent 对抗评测系统，用统一规则、统一运行环境和统一观战方式，评估不同 Agent 产出的 bot 代码能力。系统不追求一开始就做成公开赛事平台，而是先形成一个可复现、可比较、可回放的内部 benchmark arena。

目标场景不是普通网页小游戏，而是：

- 团队成员提交自己的 bot 实现
- 平台统一拉取 GitHub 仓库并构建运行
- 双方 bot 在同一张规则生成地图上对抗
- 比赛结束后输出胜负、分数、回放和统计
- 通过一个 Web 页面完成观战与回放

### 1.2 核心目标

第一版重点解决以下问题：

- 如何用一个有足够策略空间的二维回合制游戏，拉开 bot 水平差距
- 如何让 bot 接入简单，同时保证正式评测环境统一
- 如何通过 replay 和单页面观战页，让非开发者也能看懂比赛
- 如何避免固定地图被硬编码，从而更真实地评估泛化能力

### 1.3 非目标

第一版暂不覆盖以下能力：

- 公开用户注册和多租户系统
- 对外赛事平台化运营
- 大规模集群调度
- 复杂观战大厅和多页面管理后台
- 实时真直播基础设施
- 完整恶意代码扫描平台

## 2. 产品定义

### 2.1 产品定位

本产品定义为一个内部 Agent Benchmark Arena。

它有三个本质属性：

- 它是一个规则稳定的对抗环境
- 它是一个 bot 代码评测系统
- 它是一个可观战、可回放、可复盘的可视化评测台

### 2.2 一期范围

一期范围固定如下：

- 1v1 对抗
- 36x36 地图
- 每方 5 个机器人
- 全图可见
- 500 回合结束
- 地图由规则和 seed 生成
- bot 通过 GitHub 仓库地址 + Dockerfile 提交
- 平台统一构建并运行 bot
- 一个 Web 页面支持观战与回放

### 2.3 成功标准

一期上线后，至少应满足：

- 任意两个 bot 版本可以稳定发起比赛
- 相同 seed 和相同 bot commit 可复现相同结果
- 比赛结束后 5 秒内可进入回放页面
- 非开发者可以仅通过回放页面理解比赛节奏与胜负原因
- 团队内部可以把它作为 agent 策略与代码质量的对比工具

## 3. 游戏规则设计

### 3.1 设计原则

规则设计采用“高策略、低歧义”的原则。

高策略：局面足够复杂，能考验多机器人协作、路径规划、资源调度、时机控制和对手干扰。

低歧义：动作、结算顺序、冲突规则和得分规则必须非常明确，以保证 replay、调试和复盘的一致性。

### 3.2 核心设定

- 地图大小：36x36
- 对局形式：1v1
- 单方机器人数量：5
- 总回合数：500
- 可见性：全图可见
- 胜负判定：总得分高者胜

### 3.3 地图元素

地图包含以下元素：

- 能源点
- 机柜
- 障碍物
- 传送带
- 双方出生区

地图结构建议划分为五个战略区域：

- 左上资源区
- 右上资源区
- 左下资源区
- 右下资源区
- 中央枢纽区

中央枢纽区承载高冲突传送带、主要通路和至少一个高价值机柜，用于制造战略热点。

### 3.4 能源机制

地图上存在两种能源：

- E1
- E2

规则要求：

- 机器人站在能源点上时可执行 `PICK`
- 每次 `PICK` 最多获得 2 单位能源
- 机器人最大容量为 10
- 能源按波次刷新，而不是连续均匀刷新
- 建议每 25 回合进行一轮区域刷新

设计意图：

- 避免 bot 仅做最近路径贪心
- 强化中期调度和多机器人分工
- 让不同区域在不同时间段产生价值波峰

### 3.5 机柜机制

地图固定存在 3 个机柜。

每个机柜对不同能源类型有不同倍率，例如：

- 机柜 A：E1 x1.0，E2 x1.5
- 机柜 B：E1 x1.3，E2 x1.0
- 机柜 C：E1 x0.8，E2 x1.8

投递规则：

- 机器人必须站在机柜格上才能执行 `DROP`
- `DROP` 时按当前携带能源结算得分
- `DROP` 完成后机器人对应能源载荷清零

限流规则：

- 每个机柜按窗口接收能源
- 例如每 10 回合为一个接收窗口
- 每个窗口最多接收 20 单位能源
- 超额投递不计分，直到下一个窗口重新开放

设计意图：

- 避免所有 bot 永远冲同一个最优机柜
- 强化时机控制、排队决策和分流策略

### 3.6 机器人动作

每个机器人每回合只能提交 1 个动作，动作集合如下：

- `MOVE_UP`
- `MOVE_DOWN`
- `MOVE_LEFT`
- `MOVE_RIGHT`
- `PICK`
- `DROP`
- `JAM`
- `WAIT`

动作约束：

- 非法动作视为该机器人本回合 `WAIT`
- 平台记录非法动作次数

### 3.7 碰撞规则

为形成空间对抗，定义如下碰撞规则：

- 两个机器人同时尝试进入同一格：双方移动失败，原地停留
- 两个机器人尝试交换位置：双方失败，原地停留
- 目标格被静止机器人占据：移动失败
- 传送带推送目标格被占据：推送失败

设计意图：

- 形成卡位、堵路、入口争夺和机柜前对抗
- 让运营调度中带有适度空间博弈

### 3.8 传送带机制

传送带是第一版的重要高级机制。

规则如下：

- 传送带带方向
- 机器人主动进入传送带格后，在回合结算阶段自动被沿方向推送 1 格
- 传送带推送不做连锁，单回合只额外推送一次
- 若目标格非法或被占据，则推送失败

设计意图：

- 增加运输快线与风险快线
- 为 bot 提供更高阶的路径规划空间
- 在地图上塑造战术热点和 chokepoint

### 3.9 干扰机制

为了保留适度对抗性，同时避免游戏变成纯战斗游戏，引入轻量 `JAM` 机制。

规则如下：

- `JAM` 只能对相邻敌方机器人生效
- 被干扰的机器人在下一回合不能执行 `PICK` 和 `DROP`
- 被干扰机器人仍可移动
- `JAM` 有 5 回合冷却
- 满载机器人不能使用 `JAM`

设计意图：

- 让关键节点争夺更有戏剧性
- 允许 bot 打断对手的投递和采集节奏
- 控制对抗占比，保持“运营调度 60%，正面对抗 40%”

### 3.10 回合结算顺序

回合结算顺序必须固定，建议如下：

1. 收集双方所有机器人动作
2. 校验动作合法性
3. 处理主动移动
4. 处理碰撞结果
5. 处理 `PICK`
6. 处理 `DROP`
7. 处理 `JAM`
8. 处理传送带推送
9. 更新窗口、冷却和刷新逻辑
10. 记录本回合状态快照和事件流

此顺序将作为引擎实现、回放回放器、调试和争议处理的标准依据。

### 3.11 v2 规则收敛

在第一轮原型验证后，规则进一步收敛为更适合调度博弈的 `机房走廊型 v2`：

- 地图结构升级为机房走廊型
- 内墙显著增加，形成主通道、侧通道、深区和难出来区域
- 能量不再只存在于固定采集点，而是在可走路径点随机掉落
- 每个能量具有 `1-100` 的数值
- 数值采用区域偏置分布：
  - 主通道偏中低值
  - 深区偏高值
- 机器人背包限制按携带总能量值累计
- 单机器人背包上限为 `150`
- 机架不采用窗口限流
- 每个机架有总容量上限
- 第一版默认机架总容量为 `1000`

该版本的设计目标是让 bot 必须在以下问题中做权衡：

- 走稳线持续捡中低值，还是进入深区抢高值
- 背包接近上限时是否立即回仓
- 某一机架快要接近总容量上限时是否切换投递目标
- 是否为了一个高值掉落点临时改线

## 4. 地图生成设计

### 4.1 生成目标

地图不采用固定官方图池，而采用规则生成地图，以避免 bot 通过硬编码地图进行投机优化。

### 4.2 生成原则

地图生成必须满足：

- 同一 seed 下结果完全确定
- 双方出生区近似对称或镜像对称
- 能源总量与主要通路收益近似对称
- 机柜可达性近似对称
- 中央战略区必须存在可争夺价值

### 4.3 生成方式

建议采用“模板 + 参数扰动”的程序生成方式，而不是完全自由随机。

生成流程建议：

1. 选择基础模板
2. 放置出生区
3. 放置中央枢纽结构
4. 放置能源区域
5. 放置机柜
6. 放置障碍物和传送带
7. 做公平性校验
8. 产出 seed 和最终地图

这样既能保证泛化性，也能保证比赛公平和 replay 可复现。

### 4.4 seed 策略

系统需要区分两类 seed：

- 调试 seed：允许手动指定，用于复现和调试
- 评测 seed：由系统从 hidden seed 集中选择，用于正式比较

正式排行榜建议只采用 hidden seed 集合，以减少 bot 对已知地图分布的针对性优化。

## 5. Bot 接入与提交模型

### 5.1 提交方式

第一版 bot 提交方式固定为：

- 提交一个 GitHub 仓库地址
- 仓库根目录必须包含 Dockerfile

平台处理流程：

1. 记录用户提交的 repo URL
2. 解析 branch 或默认分支
3. 锁定到具体 commit SHA
4. 执行 clone + checkout
5. 执行 docker build
6. 产出可运行 bot image

正式评测必须以 commit SHA 为准，不能跟随分支漂移。

### 5.2 为什么采用 GitHub Repo + Dockerfile

该模式的优点：

- 团队接入成本低
- 运行环境统一
- bot 版本可追踪
- 便于比较 agent 不同代码版本
- 可直接把“仓库版本”映射到“评测结果”

### 5.3 运行方式

正式评测中，平台不会调用用户公网服务，而是：

- 在平台环境中启动 bot 容器
- bot 在容器内启动 HTTP 服务
- `match-runner` 通过容器网络调用 bot 接口

这样既保留多语言开放性，也保持运行公平和结果可复现。

## 6. Bot OpenAPI 契约

### 6.1 契约原则

bot 契约应满足：

- 多语言友好
- 状态描述清晰
- 每回合调用成本稳定
- 易于本地调试

### 6.2 接口列表

#### `GET /health`

用于判断 bot 服务是否启动成功。

#### `POST /init`

用于初始化比赛上下文，输入包括：

- 地图大小
- seed
- 己方阵营标识
- 机器人 id 列表
- 规则配置
- 初始地图结构

#### `POST /act`

每回合调用一次，输入包括：

- 当前回合 turn
- 当前全图状态
- 己方机器人状态
- 敌方机器人状态
- 能源点状态
- 机柜状态
- 上回合关键事件

输出包括：

- 机器人 id 到动作的映射

#### `POST /finish`

用于通知比赛结束，输入包括：

- 最终比分
- 胜负结果
- 关键统计

### 6.3 接口约束

必须固定以下约束：

- `/act` 严格超时，例如 200ms
- 超时则该回合动作视为默认 `WAIT`
- 非法动作按规则降级为 `WAIT`
- 容器默认禁外网
- bot 不可直接访问引擎内部状态

## 7. 系统架构设计

### 7.1 模块拆分

第一版建议拆成 4 个核心模块。

#### `game-engine`

职责：

- 规则状态机
- 地图生成
- 回合推进
- 事件产出
- replay 帧生成

要求：

- deterministic
- 无副作用
- 可作为纯函数式核心被 runner 驱动

#### `match-runner`

职责：

- 拉起 bot A / bot B 容器
- 调 bot 的 HTTP 接口
- 控制比赛完整生命周期
- 收集动作、耗时、异常
- 生成 replay

#### `arena-api`

职责：

- 接收 bot 提交
- 触发构建
- 创建比赛
- 提供比赛查询、排行榜、回放元数据接口

#### `web-ui`

职责：

- 加载比赛 replay
- 提供观战与回放体验
- 展示比分、事件和时间轴

### 7.2 推荐技术栈

建议栈如下：

- `game-engine`：Rust
- `match-runner`：Go
- `arena-api`：Go
- `web-ui`：Next.js + React
- 地图绘制：Canvas
- 元数据存储：Postgres
- replay / build log 存储：本地文件或对象存储

### 7.3 设计原因

Rust 适合实现高确定性、结构清晰的规则引擎。

Go 适合做 Docker 调度、GitHub 拉取、HTTP 编排和轻量后端服务。

Next.js + React 适合快速构建一个具有直播感的单页面观战与回放界面。

## 9. Replay 设计

### 9.1 Replay 的角色

Replay 是整个系统的核心资产，不只是前端播放器的数据源。

Replay 同时服务于：

- 观战
- 回放
- 复盘
- 调试
- 结果争议追溯

### 9.2 Replay 结构建议

建议 replay.json 至少包含：

- meta
  - match id
  - bot version ids
  - seed
  - map config
- initial_state
- frames[]
  - turn
  - state snapshot
  - actions
  - events
  - scores
- summary
  - winner
  - final scores
  - key stats

### 9.3 Replay-first 原则

第一版前端不要优先做实时直播，而是优先把 replay-first 做正确。

只要 replay 数据结构足够好，后面从“赛后回放”升级到“准直播”甚至“真直播”都会容易很多。

## 10. Web 页面设计

### 10.1 页面定位

第一版只做一个 Web 页面，用于比赛观战与回放。

页面定位不是后台，而是“带直播感的一页式赛事回放台”。

第一版界面语言默认为中文，以降低内部使用门槛。

### 10.2 页面结构

建议页面布局如下：

#### 顶部对阵横幅

展示：

- Bot A vs Bot B
- 双方分数
- 当前回合 / 500
- 比赛状态
- seed
- 播放控制

#### 中央主地图

展示：

- 36x36 地图
- 机器人位置、编号、阵营颜色
- 能源点和库存状态
- 机柜与窗口状态
- 传送带方向
- 障碍物
- 热点区域高亮

#### 右侧赛事信息栏

展示：

- 最近关键事件流
- 双方本局关键指标
- 机柜当前接收窗口状态
- 领先变化摘要

#### 底部时间轴

支持：

- 拖动跳转回合
- 倍速播放
- 标记关键事件节点
- 快速回到高潮片段

### 10.3 视觉方向

视觉风格采用“直播感强，但不浮夸”的竞技指挥中心风格。

设计原则：

- 地图是绝对主角
- 分数变化反馈明确
- 事件播报易懂
- 尽量避免工程细节打扰观战体验

主页面不展示：

- GitHub 仓库地址
- Docker 构建日志
- commit SHA
- 原始 HTTP 请求
- 原始 JSON 动作详情

这些工程细节应保留在后续的开发者调试页或后台接口中，而不是主观战页面。

## 11. 排名与评测设计

### 11.1 评测目标

排行榜不是只看单局输赢，而是看：

- 多个 hidden seeds 下的综合表现
- 平均得分
- 胜率
- 稳定性
- 非法动作率
- 决策耗时

### 11.2 第一版排名方案

第一版可采用简化 Elo 或 Trueskill。

推荐保留以下指标：

- rating
- wins
- losses
- avg_score
- avg_turn_latency
- invalid_action_rate

### 11.3 评测模式

建议支持三种模式：

- 单局对战
- 多 seed 批量对战
- 正式排行榜评测

正式排行榜评测只使用 hidden seed 集。

## 12. 安全与约束

第一版虽然是内部使用，仍建议保留基本运行约束：

- 容器运行
- 默认禁外网
- 固定 CPU / 内存限制
- bot 启动超时
- `/act` 请求超时
- 只允许通过 bot API 与比赛交互

原因不是做成重型安全平台，而是为了让评测结果更公平、更稳定、更少偶发因素。

## 13. MVP 实施顺序

建议按以下顺序推进：

1. 实现 `game-engine`
2. 实现本地 `match-runner`
3. 定义 bot OpenAPI 并提供 demo bot
4. 实现 GitHub repo + Dockerfile 提交流程
5. 实现 replay.json 生成
6. 实现单页面回放 UI
7. 补充排行榜和批量评测

## 14. 风险与注意事项

### 14.1 规则复杂度失控

风险：

- 规则过多会导致 bot 编写成本高
- viewer 难以解释局面

建议：

- 第一版先固定核心机制，不再继续叠加新规则

### 14.2 地图生成失衡

风险：

- 某些 seed 可能天然偏向一侧

建议：

- 地图生成后必须做公平性校验
- 正式评测采用多 seed 平均结果

### 14.3 Replay 数据过重

风险：

- 每回合全量快照可能导致 replay 体积过大

建议：

- 第一版先全量快照，确保正确
- 后续再优化为状态差量压缩

### 14.4 提交流程不稳定

风险：

- GitHub 仓库结构不规范
- Dockerfile 启动方式不统一

建议：

- 提供标准 bot 模板仓库
- 明确 Dockerfile 和 bot API 的最小规范

## 15. 结论

本设计将 Agent Arena 第一版定义为一个内部使用的、规则稳定的、回放优先的 bot 对抗评测系统。

一期不追求公开平台化，而是优先打通以下闭环：

- 提交 bot
- 平台统一构建运行
- 多 seed 对战
- 结果可复现
- Web 页面可观战与回放

这是一个足够轻、又足够有技术含量的起点，适合作为团队内部 agent 代码能力 benchmark 的基础设施。
